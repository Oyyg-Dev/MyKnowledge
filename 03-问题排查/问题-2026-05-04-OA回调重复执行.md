## 问题现象
OA 系统回调 MS 项目接口时，偶尔会出现数据重复插入的情况，导致业务数据异常。

## 环境信息
- **服务器**: MS 生产环境
- **版本**: v2.3.1
- **日志路径**: /app/logs/faret-admin.log
- **发生时间**: 2026-05-03 14:30

## 排查过程

### 1. 检查日志
```bash
# 查看错误日志
tail -f /app/logs/faret-admin.log | grep "OA回调"
```

**日志内容：**
```
2026-05-03 14:30:15 INFO  - 收到OA回调请求: requestId=12345
2026-05-03 14:30:16 INFO  - 处理OA回调成功: requestId=12345
2026-05-03 14:30:18 INFO  - 收到OA回调请求: requestId=12345
2026-05-03 14:30:19 ERROR - 数据重复插入异常: Duplicate entry
```

**发现：** 同一个 requestId 被调用了两次

### 2. 定位代码
- **相关文件**: `ums-faret/faret-admin/src/main/java/com/faret/web/controller/oa/OaCallbackController.java`
- **代码位置**: OaCallbackController.java:45

**问题代码：**
```java
@PostMapping("/callback")
public AjaxResult handleCallback(@RequestBody OaRequest request) {
    // 直接处理，没有幂等性检查
    oaService.processCallback(request);
    return AjaxResult.success();
}
```

### 3. 分析原因
1. **OA 系统重试机制**：OA 系统在网络波动时会重试回调
2. **缺少幂等性设计**：接口没有检查请求是否已处理
3. **并发问题**：两次请求几乎同时到达，都通过了检查

## 解决方案

### 临时方案
在数据库表中添加唯一索引，防止重复插入：
```sql
ALTER TABLE oa_callback_log 
ADD UNIQUE INDEX uk_request_id (request_id);
```

### 最终方案
在接口中添加幂等性检查：

**修改后的代码：**
```java
@PostMapping("/callback")
public AjaxResult handleCallback(@RequestBody OaRequest request) {
    String requestId = request.getRequestId();
    
    // 幂等性检查
    if (oaService.isRequestProcessed(requestId)) {
        log.warn("请求已处理，忽略重复调用: {}", requestId);
        return AjaxResult.success("请求已处理");
    }
    
    // 使用分布式锁防止并发
    String lockKey = "oa:callback:" + requestId;
    RLock lock = redissonClient.getLock(lockKey);
    
    try {
        if (lock.tryLock(10, 30, TimeUnit.SECONDS)) {
            // 再次检查（双重检查）
            if (oaService.isRequestProcessed(requestId)) {
                return AjaxResult.success("请求已处理");
            }
            
            // 处理业务逻辑
            oaService.processCallback(request);
            
            // 记录处理状态
            oaService.markRequestProcessed(requestId);
            
            return AjaxResult.success();
        } else {
            log.error("获取锁失败: {}", requestId);
            return AjaxResult.error("系统繁忙，请稍后重试");
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        return AjaxResult.error("处理中断");
    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```

**相关服务方法：**
```java
// OaService.java
public boolean isRequestProcessed(String requestId) {
    return oaCallbackLogMapper.selectByRequestId(requestId) != null;
}

public void markRequestProcessed(String requestId) {
    OaCallbackLog log = new OaCallbackLog();
    log.setRequestId(requestId);
    log.setProcessTime(new Date());
    log.setStatus("SUCCESS");
    oaCallbackLogMapper.insert(log);
}
```

## 经验总结

### 避免措施
1. **所有外部回调接口必须实现幂等性**
2. **使用分布式锁处理并发场景**
3. **数据库层面添加唯一约束作为最后防线**
4. **记录详细的请求日志便于排查**

### 幂等性设计要点
- 使用唯一请求 ID 标识每次请求
- 在处理前检查请求是否已处理
- 使用分布式锁防止并发问题
- 双重检查确保安全性
