# 云仓系统凌晨3点登录故障排查报告

> [!info] 故障概要
> - **故障时间**：2026年5月8日 凌晨3:00 - 4:12
> - **影响范围**：所有用户无法登录
> - **故障时长**：约72分钟
> - **根本原因**：定时任务集中启动导致资源耗尽
> - **状态**：已通过服务重启临时恢复，待实施长期方案

---

## 一、故障现象

> [!warning] 用户反馈
> 凌晨3点左右无法登录云仓系统，登录请求超时或无响应。

## 二、故障时间线

### 03:00:00 - 定时任务集中启动
- 10+个定时任务在同一时刻启动
- 包括：GetReturnDetailJob、AutoSendOrderJob、GetBillDetailJob、GetCheckReceiptJob、SyncBmItemJob 等
- 触发数百个异步任务同时执行

### 03:00:00 - 03:00:01 - 系统开始异常
- 大量"服务器内部异常"错误
- 异步方法执行失败：
  - `GetExtDocReqTempStandardComponent.multiThreadInsertDocTemp`
  - `ExtdocRetTempStandardComponent`
- 数据库查询开始堆积

### 03:18:01 - Redis 首次超时
```
Redis command timed out after 30 second(s)
```
- 同步ERP物料任务失败

### 03:19:48 - 数据库连接池耗尽 ⚠️
```
Cannot get a connection, pool error Timeout waiting for idle object
```
**这是最严重的问题点**

### 03:24:29 / 03:28:30 - Redis 持续超时
- 多次出现 Redis 命令超时
- 定时任务执行时间异常延长（正常几秒，实际执行4分钟）

### 04:11:34 - 数据库连接池管家线程严重阻塞
```
HikariPool-1 - Thread starvation or clock leap detected (housekeeper delta=34m17s)
```
- 连接池维护线程被阻塞34分钟
- 后续还有 8m20s、1m8s 的阻塞

### 04:12:40 - 系统开始恢复
- 出现正常的 HTTP 请求
- 定时任务开始正常执行
- 但仍有零星的 Redis 超时

### 08:03 - 服务重启
- WMS 服务被重启
- MySQL CPU 使用率曾高达 483%
- 系统负载从 12.40 逐渐降至 3.13

## 三、根本原因分析

### 1. 定时任务调度不合理
**问题**：10+个定时任务在凌晨3点整同时启动

**影响**：
- 行数量失控
- 系统资源瞬间耗尽

### 2. 数据库连接池配置不足
**问题**：连接池大小无法支撑高并发场景

**表现**：
- 连接池耗尽，新请求无法获取连接
- 管家线程被阻塞，无法回收空闲连接
- 导致雪崩效应

### 3. Redis 超时配置过短
**问题**：Redis 超时时间为30秒

**影响**：
- 在高负载情况下，30秒不足以完成操作
- 导致大量任务失败
- 加剧系统压力

### 4. 缺乏资源限流机制
**问题**：异步任务没有并发控制

**影响**：
- 数百个异步任务同时执行
- 数据库和 Redis 连接被快速耗尽
- 系统无法自我保护

---

## 四、对登录功能的影响

在故障期间（03:00 - 04:12），任何登录请求都会遇到：

1. **数据库连接获取失败**
   - 连接池已耗尽
   - 等待超时

2. **Redis 操作超时**
   - Session 存储失败
   - 缓存读取超时

3. **最终表现**
   - 登录请求超时
   - 页面无响应
   - 用户无法登录

## 五、解决方案

### 立即措施（已实施）

#### 1. 服务重启
- 08:03 重启 WMS 服务
- 释放被占用的资源
- 系统恢复正常

### 短期措施（需要实施）

#### 1. 错峰执行定时任务

**当前配置**：
```
所有任务：0 0 3 * * ?  (凌晨3点整)
```

**建议配置**：
```cron
GetReturnDetailJob:     0 0 3 * * ?   # 03:00
AutoSendOrderJob:       0 5 3 * * ?   # 03:05
GetBillDetailJob:       0 10 3 * * ?  # 03:10
GetCheckReceiptJob:     0 15 3 * * ?  # 03:15
SyncBmItemJob:          0 20 3 * * ?  # 03:20
GetPurchaseJob:         0 25 3 * * ?  # 03:25
其他任务:               依次间隔5分钟
```

**修改位置**：
- 数据库表：`sys_quartz_job`
- 或管理后台：系统管理 → 定时任务管理

#### 2. 增加数据库连接池大小

> [!todo] 优先级：高
> 需要重启服务生效

**配置文件**：`ilc-wms-plus_backend/src/main/resources/application.yml`

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 50        # 从20增加到50
      minimum-idle: 10             # 最小空闲连接
      connection-timeout: 30000    # 连接超时30秒
      idle-timeout: 600000         # 空闲超时10分钟
      max-lifetime: 1800000        # 最大生命周期30分钟
      leak-detection-threshold: 60000  # 连接泄漏检测60秒
```

#### 3. 增加 Redis 超时时间

**配置文件**：`ilc-wms-plus_backend/src/main/resources/application.yml`

```yaml
spring:
  redis:
    timeout: 60000               # 从30秒增加到60秒
    lettuce:
      pool:
        max-active: 50           # 最大连接数
        max-idle: 20             # 最大空闲连接
        min-idle: 5              # 最小空闲连接
        max-wait: 30000          # 最大等待时间
```

#### 4. 限制异步任务并发数

**配置文件**：`ilc-wms-plus_backend/src/main/resources/application.yml`

```yaml
spring:
  task:
    execution:
      pool:
        core-size: 10            # 核心线程数
        max-size: 50             # 最大线程数
        queue-capacity: 200      # 队列容量
        keep-alive: 60s          # 线程空闲时间
      thread-name-prefix: "Async-Service-"
```

### 长期优化措施

#### 1. 定时任务优化

**评估任务必要性**：
- 检查每个定时任务的业务价值
- 合并功能相似的任务
- 删除不再使用的任务

**分批处理大数据量任务**：
```java
// 示例：分批同步数据
public void syncDataInBatches() {
   pageSize = 1000;
    int pageNum = 1;
    
    while (true) {
        List<Data> dataList = fetchData(pageNum, pageSize);
        if (dataList.isEmpty()) break;
        
        processData(dataList);
        pageNum++;
        
        // 每批次之间休息1秒，避免压垮数据库
        Thread.sleep(1000);
    }
}
```

#### 2. 监控告警

**数据库连接池监控**：
```yaml
# 启用 HikariCP 监控
management:
  endpoints:
    web:
      exposure:
        include: health,metrics,hikaricp
  metrics:
    enable:
      hikaricp: true
```

**告警规则**：
- 数据库连接池使用率 > 80%：警告
- 数据库连接池使用率 > 90%：严重
- Redis 连接池使用率 > 80%：警告
- 定时任务执行时间 > 5分钟：警告

**监控工具**：
- Prometheus + Grafana
- 或使用云监控服务

#### 3. 数据库优化

**慢查询优化**：
- 定期检查慢查询日志
- 优化没有索引的查询
- 避免全表扫描

**索引优化**：
```sql
-- 检查缺失索引的表
SELECT 
    table_name,
    index_name,
    column_name
FROM information_schema.statistics
WHERE table_schema = 'wms_database'
ORDER BY table_name, index_name;
```

#### 4. 应用层优化

**缓存策略**：
- 对频繁查询的数据增加缓存
- 使用本地缓存（Caffeine）减少 Redis 压力
- 设置合理的缓存过期时间

**异步任务优化**：
- 使用消息队列（RabbitMQ/Kafka）削峰填谷
- 实现任务优先级队列
- 增加任务失败重试机制
