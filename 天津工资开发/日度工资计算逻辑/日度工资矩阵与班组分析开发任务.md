## 1. 入口结论

本专项已确认选择“财务 + 班长双视角”方案（方案 B）：

```text
月度日工资矩阵（财务主视图）
-> 点击单元格查看人员某天追溯
-> 班组日度分析（班长主视图）
-> 导出多 sheet 对账 Excel
```

当前不建议直接做“日工资最终发放系统”。第一版目标是解释和核对：

1. 财务能看到一个月内每个人每天拆出的工资金额，且日列合计能闭环到月度工资。
2. 班长能看到每天班组产量、工种产量、工时、支援和效率指标。
3. 用户点开某人某天金额时，能看到该金额由本班组工资、支援工资、计时工资、产量、工时和支援取高组成。

一句话边界：

```text
第一版做日度拆解和追溯，不做最终发放、不做调整金额、不做固化批次。
```

## 2. 角色视角

### 2.1 财务视角

财务最关心：

1. 月度工资总额是否和当前已完成的月度工资结果一致。
2. 每个人 1 日到 31 日每天拆出的金额是否可核对。
3. 支援工资是否能看到“支援计时工资、支援计件工资、最终取高结果”。
4. 如果某天金额异常，能否追溯到产量、工时、人员类型、支援明细和取高过程。
5. 导出的 Excel 能否接近原人工 workbook 的核对体验。

所以财务主视图必须是“月度日工资矩阵”。

### 2.2 班长视角

班长最关心：

1. 某天班组总产量是多少。
2. 哪些工种贡献了产量和产值。
3. 哪些人当天参与生产。
4. 本班组人员外出支援了多少工时。
5. 外班组人员来支援本班组多少工时。
6. 班组小时单产、人均产量、平均小时工资是否能解释。

所以班长主视图必须是“班组日度分析”，不能只给人员工资矩阵。

## 6. 数据来源和预留口径

### 6.1 已确认月度结果来源

日度矩阵的月合计必须回到当前月度工资结果：

```text
月度合计 = 本班组工资 + 支援工资 + 非整体 TIME 计时工资
```

因此，日度矩阵不是重新发工资，而是对已完成月度工资做日度拆解。

### 6.2 计件事实来源

生产计件事实建议来自：

| 表 | 作用 |
| --- | --- |
| `production_piece_bill` | 主单、日期、班组、`doc_exist_flag` |
| `production_piece_bill_job` | 工种、产量、作业班组、工种支援标识 |
| `production_piece_bill_person` | 人员、人员来源班组、支援方向、工时 |

当前代码实体已看到的字段口径：

| 当前字段 | 说明 |
| --- | --- |
| `ProductionPieceBill.docExistFlag` | `N/Y/H`，其中 `H` 为工时记录 |
| `ProductionPieceBill.classGroup` | 主单班组 |
| `ProductionPieceBillJob.erpClassNo` | 工种作业班组，缺失时可回退主单班组 |
| `ProductionPieceBillJob.isSupport` | JOB 维度是否支援工种 |
| `ProductionPieceBillPerson.supportFlag` | 人员维度支援标识 |
| `ProductionPieceBillPerson.classNo` | 人员行班组字段，需结合样例数据确认含义 |
| `ProductionPieceBillPerson.workingHours` | 工时 |

注意：当前实体里 person 行字段叫 `supportFlag`，不是 `is_support`；job 行才是 `isSupport`。

### 6.3 `DOC_EXIST_FLAG = 'H'` 工时预留

当前小程序和后端已有 `H = 工时记录` 的入口迹象，因此第一版设计必须预留工时来源。

建议口径：

| 条件 | 解释 |
| --- | --- |
| `doc_exist_flag = 'H'` | 工时记录单 |
| `supportFlag = 0` | 本班组人员在本班组工作工时，进入 HOME 工时 |
| `supportFlag = 1` | 外班组人员来支援本班组，进入被支援工时 |
| `supportFlag = 2` | 本班组人员去外班组支援，进入外出支援工时 |

### 6.4 D0 只读核实结论（2026-06-15）

已按 `ssh-db` 只读流程核实 T100 生产库结构和少量聚合样例，不写库、不改数据。

结构事实：

| 事实 | 结论 |
| --- | --- |
| 表 owner | 三张生产计件表实际在 `DSDATA` 下，查询需使用 `DSDATA.production_piece_bill*` |
| 主表字段 | `DOC_EXIST_FLAG`、`CLASS_GROUP`、`WORKING_HOURS` 存在 |
| 工种表字段 | `ERP_CLASS_NO`、`IS_SUPPORT` 存在 |
| 人员表字段 | `SUPPORT_FLAG`、`CLASS_NO`、`WORKING_HOURS` 存在 |

数据事实：

| 核实项 | 结论 |
| --- | --- |
| `doc_exist_flag='H'` 是否存在 | 存在，当前只读聚合看到 266 张 H 单，日期范围 `2025-12-01` 到 `2026-06-14` |
| 天津 `H` 样例 | 宽口径能看到少量 `FLD-TJ` H 单，班组包括 `1G4`、`3C2`、`3C8`、`3C8_Y`；但这些样例当前都是 `del_flg=1/status=0`，不是有效已审工资工时 |
| H 人员行支援标识 | 当前 H 人员样例只看到 `supportFlag=0`，未看到 `supportFlag=1/2` |
| H 人员行班组 | 当前 H 人员样例里 `person.classNo` 与 `bill.classGroup` 未看到不一致 |
| H 主表工时 | 主表 `workingHours` 多数为 0，不能作为人员日工时主来源 |
| H 人员行工时 | 人员表 `workingHours` 有值，第一版如接入 H，应以人员行工时聚合 |
| H 工种行 | 当前 H 单只看到极少工种行，不应把 H 当作产量/工种事实来源 |
| 一人一天多条 H | 存在重复人员日样例，计算必须按人员 + 日期 + 班组聚合，不能假设一人一天一行 |
| `supportFlag=2` | 当前生产样例未看到 `supportFlag=2`，所以无法验证“本班组人员去外班组支援时 `classNo` 记录外班组编码” |

开发决策：

1. 第一版可以把 `DOC_EXIST_FLAG='H'` 作为“人员日工时明细”接入候选，但当前天津工资样本没有有效已审 H 工时可参与计算；如未来启用，只能先稳定用于 `supportFlag=0` 的 HOME 工时聚合。
2. `supportFlag=1/2` 的工时方向仍保留字段和展示位置，但不要把它写成已验证计算规则。
3. 计时工资日拆仍不建议第一版强开；如要开启，先只按人员行 H 工时做权重试算，并在页面展示“工时来源：H 记录”。
4. 支援工资日落位仍应优先来自当前月度支援取高链路和计件支援明细，而不是依赖 H 工时里的 `supportFlag=1/2`。

## 7. 日度拆解公式策略

### 7.1 总原则

第一版必须满足：

```text
Σ 每日拆解工资 = 月度工资
```

因此财务矩阵优先做“月度工资日度分摊”，不是另起一套日工资发放公式。

### 7.2 支援工资

支援工资天然适合按日落位：

```text
同人同日支援计件工资 = 当天全部支援计件产值合计
同人同日支援计时工资 = 当天全部支援工时 * 支援小时价
同人同日最终支援工资 = max(支援计件工资, 支援计时工资)
```

日度矩阵中的 `supportSalary` 应直接来自同人同日取高结果。

### 7.3 本班组工资

本班组工资按班组类型分支：

| 班组/人员类型 | 第一版拆解方式 | 说明 |
| --- | --- | --- |
| 整体计件班组 | 按人员每日 HOME 有效工时权重拆月度本班组工资 | 当前采用推荐保守口径，优先贴合班组工时分母 |
| 非整体计件 `PIECE` | 按人员每日 HOME 人员单价产值权重拆月度本班组工资 | 贴合非整体计件 `PIECE` 的人员单价产值池 |
| 非整体计件 `TIME` | 第一版不拆本班组工资；展示计时池贡献和月计时工资 | 计时日工资待 `H` 工时核实后再开启日拆 |

第一版落地口径：

1. 非整体计件 `PIECE` 先按每日 HOME 人员单价产值权重拆分。
2. 支援工资按同人同日取高结果直接落日。
3. 整体计件班组按每日 HOME 有效工时权重拆月度本班组工资。

### 7.4 计时工资

非整体计件 `TIME` 的计时工资保留三种后续扩展口径，但第一版按保守口径执行：

| 方案 | 说明 | 适用条件 |
| --- | --- | --- |
| 按每日 `H` 工时权重拆 | 月计时工资按每日工时分配 | `DOC_EXIST_FLAG='H'` 工时数据可靠 |
| 按每日计时池贡献拆 | 月计时工资按 `TIME` 每日人员单价产值贡献分配 | 产值数据比工时更完整 |
| 先展示月计时工资，不拆日金额 | 日单元格只显示产值和工时，计时工资放月合计 | 第一版保守方案 |

第一版选择：先展示月计时工资，不强行拆日金额；日单元格展示产值、工时和池贡献。等 `DOC_EXIST_FLAG='H'` 数据结构和样例经只读核实后，再决定是否按每日工时权重拆计时日工资。

## 8. 导出 Excel

sheet 建议：

| sheet | 内容 |

| `人员日工资矩阵` | 财务主表，人员 × 1-31 日工资 |

### 12.5 本轮发现并修复的问题

真实库 smoke 初跑发现一个解释层筛选问题：

```text
按班组筛选时，月度日工资矩阵金额本身能闭环；
但 supportDetail / teamSummary 会混入全月所有支援数据。
```

典型表现：

| 查询范围 | 修复前支援明细行 | 修复后支援明细行 |
| --- | ---: | ---: |
| 3C1 | 594 | 115 |
| 3C2 | 594 | 64 |
| 3C8_Y | 594 | 59 |
| 3C10 | 594 | 52 |

修复口径：

1. 支援工时按 `日期 + 人员 + 工资归属班组 + 作业班组` 落在当前查询范围内。
2. 支援计件明细按 `日期 + 人员 + 工资归属班组 + 作业班组` 落在当前查询范围内。
3. `classNo` 筛选同时匹配工资归属班组和作业班组，避免“本班组人员外出支援”和“外班组支援本班组”解释丢失。
4. `userNo` 筛选同步作用到支援工时、支援计件和追溯明细。

修复后结论：

```text
矩阵金额闭环通过；
班组支援明细和班组日度统计已按查询范围收敛；
当前剩余差异只剩 0 或 0.01 元级四舍五入差异。
```
