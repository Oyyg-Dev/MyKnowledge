## 1. 入口结论

本专项已确认选择“财务 + 班长双视角”方案（方案 B）：

```text
月度日工资矩阵（财务主视图）
-> 点击单元格查看人员某天追溯
-> 班组日度分析（班长主视图）
-> 导出多 sheet 对账 Excel
```

当前不建议直接做“日工资最终发放系统”。第一版目标是解释和核对：

1. 财务能看到一个月内每个人每天拆出的工资金额，且日列合计能闭环到月度工资。
2. 班长能看到每天班组产量、工种产量、工时、支援和效率指标。
3. 用户点开某人某天金额时，能看到该金额由本班组工资、支援工资、计时工资、产量、工时和支援取高组成。

一句话边界：

```text
第一版做日度拆解和追溯，不做最终发放、不做调整金额、不做固化批次。
```

## 2. 角色视角

### 2.1 财务视角

财务最关心：

1. 月度工资总额是否和当前已完成的月度工资结果一致。
2. 每个人 1 日到 31 日每天拆出的金额是否可核对。
3. 支援工资是否能看到“支援计时工资、支援计件工资、最终取高结果”。
4. 如果某天金额异常，能否追溯到产量、工时、人员类型、支援明细和取高过程。
5. 导出的 Excel 能否接近原人工 workbook 的核对体验。

所以财务主视图必须是“月度日工资矩阵”。

### 2.2 班长视角

班长最关心：

1. 某天班组总产量是多少。
2. 哪些工种贡献了产量和产值。
3. 哪些人当天参与生产。
4. 本班组人员外出支援了多少工时。
5. 外班组人员来支援本班组多少工时。
6. 班组小时单产、人均产量、平均小时工资是否能解释。

所以班长主视图必须是“班组日度分析”，不能只给人员工资矩阵。

## 6. 数据来源和预留口径

### 6.1 已确认月度结果来源

日度矩阵的月合计必须回到当前月度工资结果：

```text
月度合计 = 本班组工资 + 支援工资 + 非整体 TIME 计时工资
```

因此，日度矩阵不是重新发工资，而是对已完成月度工资做日度拆解。

### 6.2 计件事实来源

生产计件事实建议来自：

| 表 | 作用 |
| --- | --- |
| `production_piece_bill` | 主单、日期、班组、`doc_exist_flag` |
| `production_piece_bill_job` | 工种、产量、作业班组、工种支援标识 |
| `production_piece_bill_person` | 人员、人员来源班组、支援方向、工时 |

当前代码实体已看到的字段口径：

| 当前字段 | 说明 |
| --- | --- |
| `ProductionPieceBill.docExistFlag` | `N/Y/H`，其中 `H` 为工时记录 |
| `ProductionPieceBill.classGroup` | 主单班组 |
| `ProductionPieceBillJob.erpClassNo` | 工种作业班组，缺失时可回退主单班组 |
| `ProductionPieceBillJob.isSupport` | JOB 维度是否支援工种 |
| `ProductionPieceBillPerson.supportFlag` | 人员维度支援标识 |
| `ProductionPieceBillPerson.classNo` | 人员行班组字段，需结合样例数据确认含义 |
| `ProductionPieceBillPerson.workingHours` | 工时 |

注意：当前实体里 person 行字段叫 `supportFlag`，不是 `is_support`；job 行才是 `isSupport`。

### 6.3 `DOC_EXIST_FLAG = 'H'` 工时预留

当前小程序和后端已有 `H = 工时记录` 的入口迹象，因此第一版设计必须预留工时来源。

建议口径：

| 条件 | 解释 |
| --- | --- |
| `doc_exist_flag = 'H'` | 工时记录单 |
| `supportFlag = 0` | 本班组人员在本班组工作工时，进入 HOME 工时 |
| `supportFlag = 1` | 外班组人员来支援本班组，进入被支援工时 |
| `supportFlag = 2` | 本班组人员去外班组支援，进入外出支援工时 |

当前需要确认：

1. `doc_exist_flag = 'H'` 在目标环境是否已有正式数据。
2. `supportFlag = 2` 时，`classNo` 是否记录外班组编码。
3. `classGroup`、`classNo`、`erpClassNo` 三者在 `H` 工时记录里的实际填值规则。
4. 是否存在一人一天多条 `H` 工时记录。

如果用户确认进入开发前置核实，建议用 `ssh-db` 只读查询表结构和少量样例，不写库、不改数据。

## 7. 日度拆解公式策略

### 7.1 总原则

第一版必须满足：

```text
Σ 每日拆解工资 = 月度工资
```

因此财务矩阵优先做“月度工资日度分摊”，不是另起一套日工资发放公式。

### 7.2 支援工资

支援工资天然适合按日落位：

```text
同人同日支援计件工资 = 当天全部支援计件产值合计
同人同日支援计时工资 = 当天全部支援工时 * 支援小时价
同人同日最终支援工资 = max(支援计件工资, 支援计时工资)
```

日度矩阵中的 `supportSalary` 应直接来自同人同日取高结果。

### 7.3 本班组工资

本班组工资按班组类型分支：

| 班组/人员类型 | 第一版拆解方式 | 说明 |
| --- | --- | --- |
| 整体计件班组 | 按人员每日 HOME 有效工时权重拆月度本班组工资 | 当前采用推荐保守口径，优先贴合班组工时分母 |
| 非整体计件 `PIECE` | 按人员每日 HOME 人员单价产值权重拆月度本班组工资 | 贴合非整体计件 `PIECE` 的人员单价产值池 |
| 非整体计件 `TIME` | 第一版不拆本班组工资；展示计时池贡献和月计时工资 | 计时日工资待 `H` 工时核实后再开启日拆 |

第一版落地口径：

1. 非整体计件 `PIECE` 先按每日 HOME 人员单价产值权重拆分。
2. 支援工资按同人同日取高结果直接落日。
3. 整体计件班组按每日 HOME 有效工时权重拆月度本班组工资。

### 7.4 计时工资

非整体计件 `TIME` 的计时工资保留三种后续扩展口径，但第一版按保守口径执行：

| 方案 | 说明 | 适用条件 |
| --- | --- | --- |
| 按每日 `H` 工时权重拆 | 月计时工资按每日工时分配 | `DOC_EXIST_FLAG='H'` 工时数据可靠 |
| 按每日计时池贡献拆 | 月计时工资按 `TIME` 每日人员单价产值贡献分配 | 产值数据比工时更完整 |
| 先展示月计时工资，不拆日金额 | 日单元格只显示产值和工时，计时工资放月合计 | 第一版保守方案 |

第一版选择：先展示月计时工资，不强行拆日金额；日单元格展示产值、工时和池贡献。等 `DOC_EXIST_FLAG='H'` 数据结构和样例经只读核实后，再决定是否按每日工时权重拆计时日工资。

## 9. 开发阶段拆分

### D0. 数据结构核实

目标：把 `DOC_EXIST_FLAG='H'`、`supportFlag`、`classGroup/classNo/erpClassNo` 的实际数据填值查清。

动作：

1. 只读核对表结构。
2. 只读抽查 `doc_exist_flag = 'H'` 的少量样例。
3. 区分主表班组、工种作业班组、人员行班组。
4. 输出字段事实和仍需确认项。

通过标准：

- 不再把 person 行 `supportFlag` 和 job 行 `isSupport` 混用。
- 确认 `H` 工时能否进入第一版计算。

### D1. 月度日工资矩阵

目标：先完成财务主视图。

动作：

1. 复用月度工资结果作为月合计来源。
2. 按日生成 1-31 日单元格。
3. 支援工资按同人同日取高结果落日。
4. 本班组工资和计时工资按确认后的权重拆日。
5. 返回闭环差异。

通过标准：

- 每个人 `dailyTotalSalary` 与 `monthlyTotalSalary` 可核对。
- 闭环差异为 0 或只存在四舍五入差异。
- 日期单元格可点击追溯。

### D2. 人员某天追溯抽屉

目标：解释某个日期单元格为什么是这个金额。

动作：

1. 展示本班组金额组成。
2. 展示支援金额组成。
3. 展示计时金额组成。
4. 展示产量、产值、工时和来源单据。

通过标准：

- 财务能解释某人某天金额。
- 班长能看到当天实际生产和支援情况。

### D3. 班组日度分析

目标：完成班长主视图。

动作：

1. 班组日汇总。
2. 班组工种汇总。
3. 人员工种汇总。
4. 支援明细。
5. 工时明细预留。

通过标准：

- 班长能回答某天班组产量、工时、支援和效率。
- 班组维度字段不重复塞到每个人员行。

### D4. 多 sheet 导出

目标：形成财务可保存、可复查的 Excel。

动作：

1. 导出人员日工资矩阵。
2. 导出班组日工资统计。
3. 导出人员工种明细。
4. 导出班组工种汇总。
5. 导出支援明细。
6. 导出工时与异常说明。

通过标准：

- 导出字段和页面字段能互相对应。
- 导出能解释矩阵金额。

### D5. 异常和预警

目标：避免用户把不完整数据当作最终工资。

预警项：

| 预警 | 说明 |
| --- | --- |
| 工时待接入 | 需要 `H` 工时但暂未取到 |
| 日度拆解不闭环 | 日合计与月合计存在差异 |
| 支援无计件产值 | 支援只按保底工资解释 |
| 支援无工时 | 有支援计件但无支援工时 |
| 人员类型缺失 | `OOAGUA05` 缺失 |
| 能力系数缺失 | `OOAGUA03` 缺失 |
| 单价缺失 | 工种单价缺失 |

通过标准：

- 页面和导出都能看到异常原因。
- 阻断项不允许生成正式拆解结果。
