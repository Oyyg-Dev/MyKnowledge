# 最近确认点

> 记录时间：2026-06-12  
> 作用：收口当前对话里已经定下来的天津工资口径，避免分散在多份文档里反复翻找。
> 维护顺序：口径变化先更新本页，再同步各主文档、图解和待确认页。
> 优先级：本页只做快速收口；如果和主文档冲突，以主文档为准。
> 仅保留当前已确认口径；未确认问题放在待确认页，不在这里扩写。

## 已确认

1. 非整体计件班组 `TIME` 本班组提交计件时，员工单价产值进入班组计时池，不直接发给本人。
2. 班组计时池核对优先用 `本班组车间单价产值` 和 `本班组人员单价产值`，两列导出显示保留两位小数。
3. `PIECE` 本班组明细计时池贡献 = `车间单价产值 - 人员单价产值`。
4. `TIME` 本班组明细计时池贡献 = `人员单价产值`。
5. 月度汇总和导出只对非整体计件 `TIME` 展示计时字段。
6. 近期核对样例以 `TJ生产部-冷凝器包装班` 为主。
7. P10D 天津工资计件修正只覆盖天津工资核算口径，不反写生产计件原始单据；原始计件单仍作为数量、单价、人员行、工序行、日期等底账来源。来源：本次对话确认。
8. P10D 启用修正后，工资计算优先使用启用修正记录中的工资归属班组、作业班组和 HOME/SUPPORT 场景；没有修正记录时继续使用原始计件单和月度考勤口径。来源：本次对话确认。
9. P10D 不处理“计件人员录错”的场景；如果计件单人员本身错误，应作为源头数据修正处理，不应伪装成班组或支援归属修正。来源：本次对话确认。
10. 天津工资计算存在预警项时，阻断项为 0 的前提下，前端勾选“允许预警继续计算”或接口传 `allowWarnings=true` 后可继续计算；阻断项不能通过该参数跳过。来源：本次对话确认。
11. 当前 MS 主线代码未落地 `dailySummary`、`traceDetail`、日工资报表页和日工资明细页；这些内容按后续专项处理。固化批次和月中调岗按当前决策暂不纳入未完成任务清单。来源：代码一致性回归核对 + 本次讨论。
12. 当前主线 HOME 判断只看 `WAGE_CLASS_NO == WORK_CLASS_NO`；`3C7 -> 3C7_C/3C7_Y`、`3C8 -> 3C8_B/3C8_Y` 是否仍需视为 HOME，需业务确认后再改代码。来源：代码一致性回归核对。
13. 当前代码对 `OOAGUA03` 能力系数缺失按预检阻断处理；若业务要恢复“非整体计件班组 `PIECE` 缺失默认 1.0”，需作为代码变更重新确认和验收。来源：代码一致性回归核对。
14. 天津工资知识库按“月度工资计算逻辑”和“日度工资计算逻辑”拆分；月度讨论不再混到日度底稿里，日度专项也不写成当前月度已落地能力。来源：本次讨论。
15. `支援明细取高结果 sheet` 已从当前月度遗漏项中剔除，不再作为待办；后续月度第一批只继续确认班组汇总行和班组统计指标。来源：本次讨论。
16. 当前代码完成度已按 `40-待确认/当前代码完成度核对-20260613.md` 打标：月度主线已完成预检、月工资汇总、月度导出、阻断项导出、计时工资池、支援工资取高和 P10D 代码能力；日度报表/追溯和固化批次不属于当前已完成能力。来源：本次代码核对。
17. 日度工资专项已确认第一版采用“月度日工资矩阵 + 班组日度分析 + 单元格追溯 + 多 sheet 导出”；第一版定位为日度拆解和追溯，不直接替代月度工资发放。来源：本次讨论。
18. 日度矩阵第一版按推荐口径执行：整体计件本班组日工资按每日 HOME 有效工时权重拆；非整体计件 `PIECE` 按每日 HOME 人员单价产值权重拆；非整体计件 `TIME` 先不强行拆日计时工资，只展示月计时工资和每日池贡献。来源：本次讨论。
19. `DOC_EXIST_FLAG='H'` 工时记录进入开发前置核实项：先用只读方式核实数据结构和样例，数据可靠再接入；核实前只做字段预留，不写成已确认工资数据源。来源：本次讨论。
20. 2026-06-15 已完成 `DOC_EXIST_FLAG='H'` 只读核实：目标 T100 生产库存在 H 工时记录和少量天津样例；当前 H 人员行只看到 `supportFlag=0`，`classNo` 与主表 `classGroup` 未见不一致，未看到 `supportFlag=1/2`；第一版可把 H 作为 HOME 人员日工时候选来源，支援方向继续预留。来源：D0 只读核实。
21. 2026-06-15 已在 worktree 分支 `feature/tianjin-daily-wage-matrix` 完成日度工资矩阵代码初版：后端新增 7 个日度接口和多 sheet 导出，前端新增 `workshop/tianjinSalary/daily/index` 页面；后端编译和前端生产构建已通过。来源：本次开发实现。
22. 日度矩阵第一版的 `dailyTotalSalary` 口径调整为“日拆 + 月结合计”：日列金额加回非整体 `TIME` 暂不拆日的计时月结金额，再与 `monthlyTotalSalary` 比较闭环差异，避免把已确认月结展示的计时工资误判成异常。来源：本次开发实现。

## 待确认

1. 日度工资第一版是否只展示 `TIME` 的产值，不直接出金额。
2. 如果手工核对仍对不上，先查是否混入非 HOME 明细、支援口径或源数据异常，不先改公式。
3. P10D 相关前端、后端和修正表是否已经正式部署到生产环境。
4. “允许预警继续计算”是否需要在生产页面查询区常驻显示，还是保持在预检结果区展示。
5. 计件人员录错的源头修正是否需要单独审批、留痕表或由现有生产计件维护流程负责。
6. 整体计件班组是否也必须强制维护 `OOAGUA05=PIECE/TIME`。
7. `supportFlag=1/2` 在 H 工时记录中的真实样例，以及 `supportFlag=2` 时 `classNo` 是否记录外班组编码。
8. 整体计件班组日工资按 HOME 有效工时权重拆后，财务是否还需要一个按 HOME 车间单价产值权重的对照列。
9. 日度工资矩阵页面是否现在进入菜单配置和联调验收；推荐路径为 `workshop/tianjinSalary/daily/index`，位置放在“车间管理 / 工资核算”下。
