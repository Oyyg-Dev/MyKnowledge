### 查询规则（重要！）

⭐ **必须使用完整表名**：`DSDATA.表名`

```sql
-- ✅ 正确写法
SELECT * FROM DSDATA.ooag_t WHERE ROWNUM <= 10;

-- ❌ 错误写法（会查询 T100_RO/T100_RO_PROD 模式，为空）
SELECT * FROM ooag_t WHERE ROWNUM <= 10;
```

### 切换当前模式（推荐）

```sql
-- 切换到 DSDATA 模式
ALTER SESSION SET CURRENT_SCHEMA = DSDATA;

-- 之后可直接写表名
SELECT * FROM ooag_t WHERE ROWNUM <= 10;
```

⚠️ **注意**: DS 模式是 T100 设计器的元数据，与日常业务查询无关，不要查询。

## 🔧 核心架构认知

### 1. 代码载体优先看 `.tap`

T100 设计器"产出程序"时会读取：
- `.tap`（客制段 XML）
- `.tgl`（画面架构）
- 再生成 `.4gl`

因此：
- ❌ 直接改 `.4gl` 不能替代 `.tap` 点位修改
- ✅ 涉及设计器客制段时，必须优先定位 `.tap` 中对应 `<point>` 的 `CDATA`

### 2. 逻辑必须分层

即使代码都在单个 `.4gl` 文件中，也必须逻辑分层：

- **UI 交互区**: 只做 `DIALOG / INPUT / DISPLAY / ON ACTION`
- **业务逻辑区**: 做规则校验、状态流转
- **数据访问区**: 做 SQL、游标、事务封装

🔴 **核心红线**: **禁止在 `ON ACTION` 中从弹窗直接写到事务和提交。**

### 3. 设计器点位保持轻量

设计器 Add-point 里尽量只保留一行 `CALL`，复杂逻辑放在外部独立 `.4gl` 文件中。

## 💻 工作流程

### 当前主链路

```text
收到需求（已明确作业号与环境）
  → sync_pull (更新 baseline .tzc，按需维护 working)
  → 阅读 working/.4gl / working/.tap
  → server_program_tools analyze（按需）
  → 修改 working
  → precheck
  → SSH / SQL / 日志 / 人工场景验证
  → pack_tzc.sh 打包成 .tzc
  → 输出到 code/current/<程序名>(c).tzc
  → 用户用 T100 设计器导入 .tzc，手动 diff 后保存
  → 更新 context/5-history/开发记录.md
```

### 注意事项

- ✅ 这是**本地分析与验证辅助链路**
- ❌ 不是自动部署链路
- ✅ **AI 负责修改代码并打包成 .tzc**
- ✅ **用户只关注 code/current/ 目录下的 .tzc 文件**
- ✅ **用户用 T100 设计器导入 .tzc，手动 diff 对比后保存**
