# 创新平台 - 接口与数据库设计文档
## 一、权限设计方案（细粒度权限控制）
### 1.1 角色定义
| 角色代码 | 角色名称 | 权限范围 | 说明 |
| --- | --- | --- | --- |
| `employee` | 普通员工 | 提交提案、查看自己的提案、收藏、评论、点赞 | 基础权限 |
| `dept_leader` | 部门负责人 | 员工权限 + 查看本部门提案、推荐优质提案 | 仅能访问本部门数据 |
| `project_manager` | 项目经理 | 员工权限 + 项目管理、任务分配、跨部门协作 | 可管理参与的项目 |
| `innovation_admin` | 创新管理员 | 查看全部提案、数据分析、案例管理、知识库管理 | 平台运营管理 |
| `admin` | 系统管理员 | 全部权限 + 系统配置、用户管理、权限分配 | 最高权限 |


**说明：**

+ 审批流程（部门经理、评审专家、财务）都在OA系统完成，小程序无需相关权限
+ 小程序主要负责：提案提交、查看进度、项目管理、数据展示
+ 一个用户可拥有多个角色（如：既是员工又是部门负责人）

#### 1.3.4 部门数据隔离
**原则：**

1. **部门负责人**只能查看和管理本部门的数据
2. 部门字段基于用户表的 `dept` 字段匹配
3. 后端接口自动过滤，前端无需额外处理

**实现示例：**

```javascript
// 后端接口示例
function getProposalList(userId, userRole, userDept, filters) {
  let query = db.proposals
  
  if (userRole === 'dept_leader') {
    // 部门负责人只能看本部门
    query = query.where('dept', '=', userDept)
  } else if (userRole === 'employee') {
    // 普通员工只能看自己的
    query = query.where('created_by', '=', userId)
  }
  // project_manager, innovation_admin, admin 可看全部
  
  return query.where('status', '!=', 'Draft').get()
}
```

```javascript
// 前端权限检查示例（小程序端）
const permissions = {
  // 提案查看权限
  canViewProposal(proposal, currentUser) {
    // 1. 管理员可查看全部
    if (['admin', 'innovation_admin'].includes(currentUser.role)) return true
    
    // 2. 提案作者可查看
    if (proposal.createdBy === currentUser.id) return true
    
    // 3. 部门负责人可查看本部门的（非草稿）
    if (currentUser.role === 'dept_leader' && 
        proposal.dept === currentUser.dept && 
        proposal.status !== 'Draft') {
      return true
    }
    
    // 4. 项目经理可查看已提交的
    if (currentUser.role === 'project_manager' && proposal.status !== 'Draft') {
      return true
    }
    
    // 5. 项目成员可查看已立项的
    if (proposal.projectId && proposal.projectMembers?.includes(currentUser.id)) {
      return true
    }
    
    return false
  },
  
  // 提案编辑权限
  canEditProposal(proposal, currentUser) {
    // 管理员可编辑全部
    if (currentUser.role === 'admin') return true
    
    // 只有作者可编辑，且状态为草稿或已驳回
    return proposal.createdBy === currentUser.id && 
           ['Draft', 'OARejected'].includes(proposal.status)
  },
  
  // 提案删除权限
  canDeleteProposal(proposal, currentUser) {
    // 管理员可删除全部
    if (currentUser.role === 'admin') return true
    
    // 只有作者可删除草稿
    return proposal.createdBy === currentUser.id && proposal.status === 'Draft'
  },
  
  // 项目立项权限
  canCreateProject(proposal, currentUser) {
    // 项目经理、创新管理员或管理员可立项，且提案已通过OA审批
    return ['project_manager', 'innovation_admin', 'admin'].includes(currentUser.role) && 
           proposal.status === 'OAApproved'
  },
  
  // 项目查看权限
  canViewProject(project, currentUser) {
    // 管理员可查看全部
    if (['admin', 'innovation_admin', 'project_manager'].includes(currentUser.role)) {
      return true
    }
    
    // 部门负责人可查看本部门的
    if (currentUser.role === 'dept_leader' && project.dept === currentUser.dept) {
      return true
    }
    
    // 项目成员可查看
    if (project.teamMembers?.includes(currentUser.id)) {
      return true
    }
    
    return false
  },
  
  // 项目编辑权限
  canEditProject(project, currentUser) {
    // 管理员可编辑全部
    if (['admin', 'innovation_admin'].includes(currentUser.role)) return true
    
    // 项目负责人可编辑
    if (currentUser.role === 'project_manager' && project.ownerId === currentUser.id) {
      return true
    }
    
    return false
  },
  
  // 任务查看权限
  canViewTask(task, currentUser) {
    // 管理员可查看全部
    if (['admin', 'innovation_admin'].includes(currentUser.role)) return true
    
    // 项目经理可查看项目内的
    if (currentUser.role === 'project_manager' && 
        task.project?.ownerId === currentUser.id) {
      return true
    }
    
    // 部门负责人可查看本部门成员的
    if (currentUser.role === 'dept_leader' && 
        task.assignee?.dept === currentUser.dept) {
      return true
    }
    
    // 任务负责人可查看
    if (task.assigneeId === currentUser.id) return true
    
    // 项目成员可查看
    if (task.project?.teamMembers?.includes(currentUser.id)) return true
    
    return false
  },
  
  // 任务编辑权限
  canEditTask(task, currentUser) {
    // 管理员可编辑全部
    if (['admin', 'innovation_admin'].includes(currentUser.role)) return true
    
    // 项目负责人可编辑项目内的
    if (currentUser.role === 'project_manager' && 
        task.project?.ownerId === currentUser.id) {
      return true
    }
    
    // 任务负责人可编辑自己的
    if (task.assigneeId === currentUser.id) return true
    
    return false
  },
  
  // 数据导出权限
  canExportData(scope, currentUser) {
    // 管理员和创新管理员可导出全部
    if (['admin', 'innovation_admin'].includes(currentUser.role)) return true
    
    // 部门负责人可导出本部门数据
    if (currentUser.role === 'dept_leader' && scope === 'department') return true
    
    // 项目经理可导出项目数据
    if (currentUser.role === 'project_manager' && scope === 'project') return true
    
    return false
  },
  
  // 敏感字段可见性
  canViewSensitiveField(fieldName, data, currentUser) {
    const sensitiveFields = ['budget', 'costSaving', 'reviewScore', 'contactInfo']
    
    if (!sensitiveFields.includes(fieldName)) return true
    
    // 管理员可查看全部
    if (['admin', 'innovation_admin'].includes(currentUser.role)) return true
    
    // 项目经理可查看全部
    if (currentUser.role === 'project_manager') return true
    
    // 部门负责人可查看本部门的
    if (currentUser.role === 'dept_leader' && data.dept === currentUser.dept) {
      return true
    }
    
    // 数据拥有者可查看
    if (data.createdBy === currentUser.id || data.ownerId === currentUser.id) {
      return true
    }
    
    return false
  }
}

// 注意：审批操作在OA系统完成，小程序只负责查看审批进度

// 前端使用示例
function renderProposalActions(proposal, currentUser) {
  const actions = []
  
  if (permissions.canViewProposal(proposal, currentUser)) {
    actions.push({ label: '查看详情', action: 'view' })
  }
  
  if (permissions.canEditProposal(proposal, currentUser)) {
    actions.push({ label: '编辑', action: 'edit' })
  }
  
  if (permissions.canDeleteProposal(proposal, currentUser)) {
    actions.push({ label: '删除', action: 'delete' })
  }
  
  if (permissions.canCreateProject(proposal, currentUser)) {
    actions.push({ label: '立项', action: 'createProject' })
  }
  
  return actions
}
```

### 1.6 权限验证流程
#### 1.6.1 后端权限验证
```javascript
// 后端中间件示例
function checkPermission(requiredRole, requiredPermission) {
  return async (req, res, next) => {
    const user = req.user // 从 token 中解析
    
    // 1. 检查角色
    if (requiredRole && !user.roles.includes(requiredRole)) {
      return res.status(403).json({ 
        code: 403, 
        message: '权限不足' 
      })
    }
    
    // 2. 检查具体权限
    if (requiredPermission && !user.permissions.includes(requiredPermission)) {
      return res.status(403).json({ 
        code: 403, 
        message: '权限不足' 
      })
    }
    
    next()
  }
}

// 使用示例
router.delete('/api/proposals/:id', 
  checkPermission('admin', 'proposal:delete'),
  deleteProposal
)
```

#### 1.6.2 前端权限验证
```javascript
// 前端权限管理
class PermissionManager {
  constructor(userInfo) {
    this.user = userInfo
  }
  
  // 检查是否有某个角色
  hasRole(role) {
    return this.user.roles?.includes(role)
  }
  
  // 检查是否有某个权限
  hasPermission(permission) {
    return this.user.permissions?.includes(permission)
  }
  
  // 检查是否是数据拥有者
  isOwner(data) {
    return data.createdBy === this.user.id || 
           data.ownerId === this.user.id
  }
  
  // 检查是否是同部门
  isSameDept(data) {
    return data.dept === this.user.dept
  }
}

// Vue 组件中使用
const pm = new PermissionManager(currentUser.value)

const canEdit = computed(() => {
  return pm.hasRole('admin') || 
         (pm.isOwner(proposal.value) && 
          ['Draft', 'OARejected'].includes(proposal.value.status))
})
```

## 五、前端调用示例
### 5.1 API封装（uni-app）
```javascript
// utils/request.js
const BASE_URL = 'https://api.example.com'

function request(options) {
  const token = uni.getStorageSync('token')
  
  return new Promise((resolve, reject) => {
    uni.request({
      url: BASE_URL + options.url,
      method: options.method || 'GET',
      data: options.data,
      header: {
        'Content-Type': 'application/json',
        'Authorization': token ? `Bearer ${token}` : ''
      },
      success: (res) => {
        if (res.data.code === 200) {
          resolve(res.data.data)
        } else if (res.data.code === 401) {
          uni.showToast({ title: '请先登录', icon: 'none' })
          uni.navigateTo({ url: '/pages/login/login' })
          reject(res.data)
        } else {
          uni.showToast({ title: res.data.message, icon: 'none' })
          reject(res.data)
        }
      },
      fail: (err) => {
        uni.showToast({ title: '网络错误', icon: 'none' })
        reject(err)
      }
    })
  })
}

export default request
```

## 六、总结
### 6.1 接口统计
+ 用户认证：3个
+ 提案管理：12个
+ 点赞和浏览：4个
+ OA流程同步：2个（原审批评审模块调整）
+ 项目管理：8个
+ 数据统计：4个
+ 消息通知：3个
+ 文件上传：1个
+ **合计：37个接口**

**说明：**

+ 减少了3个审批评审接口（审批/驳回/评审），因为这些操作在OA系统完成
+ 增加了4个点赞和浏览接口，用于成果案例的互动统计
+ 小程序端负责：提案提交、查看OA进度、项目管理、数据展示、点赞浏览

### 6.2 数据库表统计
+ 核心业务表：6张（用户、提案、审批、评审、项目、任务）
+ 关联关系表：7张（附件、里程碑、团队、收藏、评论、浏览记录、点赞记录）
+ 系统支持表：2张（消息、配置）
+ **合计：15张表**

**说明：**

+ `t_approval_flow` 和 `t_review` 表主要用于同步OA数据，供小程序查看
+ 审批和评审操作的数据录入由OA系统负责，小程序后端定时同步
+ 新增 `t_view_record` 和 `t_like_record` 表支持点赞和浏览功能

### 6.3 开发建议
1. 使用JWT进行身份认证
2. 接口添加接口限流（防止恶意调用）
3. 敏感数据加密存储（密码、个人信息）
4. 文件上传使用云存储（OSS/COS）
5. **【重要】定时任务同步OA审批状态**（建议每5分钟同步一次，或使用OA webhook实时推送）
6. 使用Redis缓存热点数据（提案列表、统计数据）
7. 数据库读写分离提升性能
8. 接口文档使用Swagger自动生成
9. **OA系统集成方式：**
    - 方式1：定时任务轮询OA接口获取审批状态
    - 方式2：OA系统通过Webhook回调通知审批结果（推荐）
    - 提案提交后，将提案数据推送到OA系统，获取OA流程ID
10. 小程序端只展示OA审批进度，不进行审批操作
